# Live Agent — как се пише от името на VBM канала и какви са опциите

Дата: 15.07.2026 | Статус на акаунта: Infobip Conversations **не е активен** (API връща 403)

---

## Ситуацията накратко

Преди сме ползвали **Infobip Conversations** — готовият agent inbox в Infobip портала, където
входящите съобщения влизат в опашка и агентът отговаря директно от уеб интерфейса на Infobip.
Това е най-user-friendly вариантът, но **в момента не е наличен на този акаунт** — Conversations
API (`/ccaas/1/*`) връща `403 UNAUTHORIZED`, т.е. модулът не е включен (или API ключът няма scope за него).

Затова живата комуникация в момента минава през **нашия собствен dashboard** (custom решение,
което покрива същия сценарий).

---

## Как работи сега (нашето решение)

```
Клиент пише във Viber (VBM или Bot)
        │
        ▼
Infobip → POST /webhook (нашият backend)
        │
        ├─ agent_mode ИЗКЛЮЧЕН → AI отговаря автоматично
        │       └─ клиентът напише "искам агент" → agent_mode ВКЛЮЧЕН + SMS до оператора
        │
        └─ agent_mode ВКЛЮЧЕН → съобщението чака човек в Agent Queue
```

### Начини да пишеш от името на VBM (сендър TCP)

| Начин | Кога | Колко е лесно |
|---|---|---|
| **Dashboard → Agent Queue** | Клиентът е в agent_mode | Най-лесно: чат изглед, пишеш и пращаш |
| **POST `/agent-reply`** (наш API) | По всяко време, за произволен клиент | curl/Postman с Bearer token |
| **Директно Infobip Messages API** | По всяко време | curl, но НЕ се записва в историята — избягвай |

**`/agent-reply` пример:**
```bash
curl -X POST https://<app-url>/agent-reply \
  -H "Authorization: Bearer <session-token>" \
  -H "Content-Type: application/json" \
  -d '{"to": "3598881234567", "text": "Здравейте!", "release_agent_mode": false}'
```
- Каналът (VBM/Bot) се определя автоматично от разговора.
- Изпращането включва agent_mode (AI спира). `"release_agent_mode": true` връща клиента на AI.

### Ограничението на VBM (важи за всички варианти)

Свободен текст (не-template) към VBM може да се праща **само в session прозореца** —
до 24 часа след последното съобщение **от клиента**. Извън прозореца Infobip отказва
free-form текст и трябва да се ползва регистриран template (Notify панела).
Затова агентският flow винаги е „отговор“ — не можеш да започнеш свободен разговор пръв.

## Опция: връщане към Infobip Conversations (готовият инбокс)

### Какво дава
- Готов уеб интерфейс за агенти в Infobip портала — без наша поддръжка на UI
- Опашки, разпределение между няколко агента, статуси, бележки, тагове
- Мобилно приложение за агенти

### Какво коства
1. **Трябва да се поиска активация** от Infobip (account manager) — модулът е платен add-on.
2. **Роутингът е или-или**: входящите съобщения отиват или в Conversations, или в нашия
   webhook. Ако се насочат към Conversations:
   - нашият AI асистент спира да вижда съобщенията (освен ако не се вгради като
     chatbot през Infobip Answers + bot-to-agent handover);
   - нашият dashboard (история, статистики, notify контекст) остава сляп за разговорите.
3. Хибрид е възможен, но по-сложен: AI-ят се мести в Infobip Answers, а handoff-ът
   става с трансфер към Conversations агент. Това е отделен проект.

### Извод

| | Нашият dashboard (сега) | Infobip Conversations |
|---|---|---|
| Цена | 0 (собствен код) | Платен модул |
| AI асистент с наш контекст | ✅ вграден | ⚠️ само през Answers (миграция) |
| Готов агентски UI | ⚠️ базов (Agent Queue) | ✅ пълноценен |
| Много агенти/опашки/тагове | ❌ | ✅ |
| Контрол върху данните/логиката | ✅ пълен | ⚠️ ограничен |

**Препоръка:** при 1-2 агента и нужда от AI с контекст върху изпратените известия —
оставаме на нашето решение и при нужда доразвиваме dashboard-а (напр. поле за отговор
директно в „Разговори“, без да чакаме agent_mode). При кол-център с няколко агента —
искане до Infobip за активиране на Conversations + миграция на AI-я в Answers.

## Какво може да се подобри в нашия dashboard (ако останем на него)

- [ ] Поле за отговор в ConversationDetail — писане към всеки клиент без agent_mode
- [ ] Browser нотификация/звук при нов клиент в Agent Queue (сега е SMS + 8s полинг)
- [ ] Индикатор „в session прозорец / изтекъл“ за всеки разговор (може ли free-form или само template)
- [ ] Брояч на непрочетени съобщения per разговор
